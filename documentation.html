1 Introduction

1.1 Purpose

The product described in this documentation represents the frontend part of a bigger project named AuCo(Autograph Collector). This SRS encapsulates version 1.0 of this product, which is composed of HTML, CSS and JavaScript files.

1.2 Document Conventions

This document is written using Times New Roman. Priorities are indicated by the <h1> - <h6> tags, with the highest one being represented by the <h1> tag. The content is structured in different sections using the "Scholarly HTML" format. Each section has its own number in accordance with the priorities mentioned above.

1.3 Intended Audience and Reading Suggestions

The documentation addresses the project manager and the developers involved. It describes the product, the requirements and the implemented features. The most pertinent sections for the project manager are 1.4 Product Scope and 2.3 User Classes And Characteristics. The sections regarding code implementation, and the most relevant for the developers, are 2.2 Product Functions, 2.5 Design And Implementation Constraints and 4 System Features.

1.4 Product Scope

AuCo is a useful tool for amateur autograph collectors, that allows users to organize their acquired autographs and store their unique specifications and descriptions. This website helps collectors access their own autograph database easily and gain insight on some global statistics. AuCo uses a personal system of estimating values within the app, and every autograph has a value expressed in auco points. Also, users can access their own statistics in order to see information like their total value acquired in points.

2 Overall Description

2.1 Product Perspective

This product is part of the collection-organizing-apps family, and gives the user an overview of their own collection of autographs in the "Statistics" section.

2.2 Product Functions

  1. User Interface:
    1. Creating an account
    2. Adding an autograph
    3. Editing the profile
    4. Searching through the autographs
    5. Changing settings
    6. Account personalization
    7. Seeing personal and global statistics
  2. AuCo system
    1. Database storing system

2.3 User Classes and Characteristics

This product is destined to all classes of users. The autograph collection community has no age limit. The main characteristic of the users is having a valid email address. Also, the average user will show enthusiasm regarding collecting autographs and storing them. Even a person with just one autograph can find AuCo really helpful. Every AuCo functionality is accessible for all the users.

2.4 Operating Environment

The product can optimally operate in any web browser that supports HTML5.

2.5 Design and Implementation Constraints

The main constraint that AuCo imposes on the user is a superior limit in reference to the possible number of autographs stored. A single profile can store up to 2000 entries.

2.6 User Documentation

The AuCo website comes with self-explanatory functionalities, and all of them are indicated by specific buttons with suggestive names like "Log out", "See your stats" or "New autograph".

2.7 Assumptions and Dependencies

This product has two dependencies. The main one is represented by the chosen hosting server and the other one consists of the database used for storing all the user data. Any problem that occurs regarding the database or the host provider that threatens this product can be solved by switching to another third party provider. In addition, any memory issues that may occur will determine AuCo to expand the available storage with the help of future investments.

3 External Interface Requirements

3.1 User Interfaces

When the user access the website there is displayed a welcome page that lets the user either log in into an existing account or create a new one by accessing the register page. In the main page the user has the option to search for a specific autograph, or they can add a new one. The autographs are displayed as buttons that show an image of the autograph and a short description when clicked on. The user also has the option to access a page of statistics where he can gain useful information about either their or others' autographs. If wanted, the user can log out of the account or edit their information and account settings.

3.2 Hardware Interfaces

This website is supported by all devices on which the user can access a browser. The chosen protocol used for communication is HTTPS and the data will be transferred via either JSON or CSV format.

3.3 Software Interfaces

The website will use ORACLE as a third party provider of a database, which will communicate with AuCo through an API, using JSON and CSV formats.

3.4 Communications Interfaces

The website will use different APIs in order to communicate with the database, and will send the information via JSON and CSV files, and to send the confirmation emails. AuCo will use HTTPS as a communication standard. Sensitive and personal information will be encrypted for security issues.

4 System Features

4.1 Account Management

4.1.1 Description and Priority

This feature includes creating an account by registering, logging in with an existing account and editing your current profile. The AuCo verification system send a confirmation email during registering. This is a high priority feature because the user can only access the website through an account.

4.1.2 Stimulus/Response Sequences

4.1.2.1 Register

This feature is triggered by the user after clicking the "Register" button, which sends them to the register page. After completing the form and pressing the "Submit" button, the verification process begins. The website checks if the email is valid, if the password meets the safety requirements and matches the confirmation password and if the username is available. Next, the user data is sent to the database and updated using JSON and CSV files and a communication API. After this step, the user receives a confirmation on the website and on their email address.

4.1.2.2 Login

This feature is triggered by the user after clicking the "Login" button, which sends them to the login page. After completing the form and pressing the "Login" button, the verification process begins. The website checks if the entry is found within the database. If the user submitted a valid entry, they will be sent to the main page.

4.1.2.3 Profile Editing

This feature is triggered by the user after clicking the "Edit Profile" button, which sends them to the "Edit Profile" page. After completing the form and pressing the "Submit" button, the verification process begins. If the user modified their password, the website will verify it in conformity with section 4.1.2.1 Register. If the changes are valid, the website updates the data within the database.

4.1.3 Functional Requirements

In order to successfully use this feature, the user needs a valid and accessible email address.

4.2 Autograph Management

4.2.1 Description and Priority

This feature includes uploading new autographs to the website's database, searching for an existing one and deleting unwanted autographs. This is a high priority feature because it describes the main point of the website.

4.2.2 Stimulus/Response Sequences

4.2.2.1 Adding a New Autograph

This feature is triggered by the user after clicking the "New Autograph" button, which triggers the interfaces' response. Next, the user has to provide the required specifications and upload a photo, which are added to the database.

4.2.2.2 Searching for an Autograph

This feature is triggered by the user after entering some tags in the searchbar. The website provides the matching elements after searching the database.

4.2.2.3 Delete

This feature is triggered by the user after clicking the "Delete" button within an autograph element. The website will ask for confirmation and by accepting, the database will be updated, and feedback will be shown.

4.2.3 Functional Requirements

In order to successfully use this feature, the user needs a valid account.

4.3 Seeing Statistics

4.3.1 Description and Priority

This feature lets the user see his own statistics and the global ones displayed. This is not a very high priority feature because the main scope of the website is achievable without it, but it can get the users to be more involved.

4.3.2 Stimulus/Response Sequences

This feature is accessible for the user in the "Statistics" page, which displays the information from the database.

4.3.3 Functional Requirements

In order to successfully use this feature, the user needs a valid account.
